# `excel-com` 0.1.0 release notes

`excel-com` 0.1.0 is the first public release of an experimental,
Windows-only Excel desktop Automation crate. It establishes typed COM wrappers
for application/session ownership, workbooks, worksheets, ranges, values,
formulas, formatting, tables, charts and drawings, text transformation,
external data, and PivotTables.

## Stability and compatibility

The public API is experimental. Version 0.1 establishes its initial shape, but
breaking changes remain possible before 1.0. Rust 1.85 is the MSRV. Windows
desktop Excel through COM Automation is supported; x64 is compile- and
runtime-tested and x86 is compile-tested. Excel version, printer, and external
provider differences can affect individual operations.

## Ownership and safety

Create an `OwnedApplication` for a crate-started Excel server; only it can
call `quit`. `AttachedApplication` has no shutdown method and never closes
unrelated workbooks. Wrappers are COM-apartment-bound and neither `Send` nor
`Sync`. Public APIs preserve Automation values without raw pointers and redact
sensitive diagnostic values. `macro-runtime` is opt-in and enables
`Application::run_macro`; it should be enabled only for trusted workbooks.

## Known limitations

* Windows desktop Excel only; no events, VBA source editing, ActiveX/Form
  controls, Data Model/DAX mutation, Office Scripts, or Graph workbook API.
* COM requires a creating STA thread and appropriate caller message pumping.
* External data, Power Query, and connection refresh are provider/account
  dependent; only bounded inspection and refresh orchestration are supplied.
* Some modern chart options are absent from Excel's classic COM model.
* PageSetup and fixed-format export can depend on installed printers/filters.
* Some operations require a visible or active Excel state.
* The current development host cannot reliably execute `Workbooks.Add` and
  returns `0x800A03EC`, reproduced by an independent tool. This does not
  remove the public API or block release preparation; copied fixtures use
  `Workbooks.Open` instead. Verify Add on another Windows/Excel installation.

No crate was published, no GitHub release or tag was created, and no Office
configuration was modified while preparing these notes.
