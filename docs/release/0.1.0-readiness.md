# `excel-com` 0.1.0 release-readiness report

## Scope and commits

* Release target: `excel-com` 0.1.0, experimental and Windows-only.
* Starting `origin/master`: `b30337fa4d67f0deee0060adf152ed232a41722f`.
* Ending commit: recorded by the release-preparation pull request after this
  report is committed.
* Feature freeze: active; this preparation adds no object-model domain.

## Public API and compatibility

The complete all-feature API baseline is
[`0.1.0-public-api.txt`](0.1.0-public-api.txt). Canonical domain modules now
make specialized APIs discoverable while the reviewed root convenience facade
remains frozen for 0.1. The pre-release breaking corrections remove
`Application::new`, raw message-filter/retry-classification exports, and the
obsolete `XlSheetVisibility` alias. `OwnedApplication::new` is the sole
crate-created session constructor; attached sessions never expose shutdown.

Collections use one-based Excel indexing; typed iterators are fallible and
fused. `AutomationValue` preserves Excel states and `MixedValue` represents
mixed formatting. `ExcelComError` is the primary error; `InvocationError`
retains stable member/HRESULT/EXCEPINFO accessors without raw ABI details.
The sole opt-in feature is `macro-runtime`; it is disabled by default.

MSRV is Rust 1.85. The x86_64 Windows target compiled successfully and the
fixture smoke test ran successfully on this x64 host. `i686-pc-windows-msvc`
compiled successfully after installing its Rust target; no 32-bit Excel runtime
claim is made.

## Safety audit

The audit records 62 unsafe blocks and 22 unsafe/extern declarations in the
private COM kernel. Each has a local `SAFETY:` justification. `VARIANT`,
SAFEARRAY, BSTR, COM-reference, handle, Drop, and redaction rules are reviewed
in [`0.1.0-unsafe-audit.md`](0.1.0-unsafe-audit.md). Static assertions cover
the representative wrapper set and four trybuild cases verify `!Send`,
attached-session shutdown denial, option borrowing, and consuming close.

## Known limitations and recommendation

Known limitations are authoritative in the release notes: Windows desktop
Excel/STA only; no events, VBA source editing, ActiveX/Form controls or
Data-Model/DAX mutation; provider-dependent external data; classic-object-model
chart limits; printer-dependent output; and Excel version differences.

`Workbooks.Add` is not a blocker. This host reliably reproduces a
machine-specific `0x800A03EC` Add failure also seen by an independent tool, so
the release smoke test uses copied fixtures and `Workbooks.Open`. Verify Add on
another Windows/Excel installation.

**Recommendation: Ready for 0.1.0 with documented limitations**, contingent
only on the final clean package and `cargo publish --dry-run` checks. No crate
publication, release, tag, merge, Office configuration change, process
termination, attached-session quit, raw-pointer export, or historical-evidence
rewrite occurred.
