# pywin32 versus comtypes

Generated by `excel-com-client-kb`; do not edit by hand. This is source-derived client-implementation evidence, not Excel runtime evidence unless explicitly labelled Control-confirmed.

Shared, source-supported rules:

- Automation positional arguments are reversed on `IDispatch::Invoke` paths.
- Property puts carry `DISPID_PROPERTYPUT` on `IDispatch` paths.
- Both use LCID 0 by default for their exposed dispatch helpers.
- Both translate `None` to `VT_NULL`, distinguish an explicit missing marker, and wrap object returns.
- Both own temporary `VARIANT`/`DISPPARAMS` storage and translate failing HRESULTs to Python exceptions.

Implementation-specific rules:

- pywin32 makepy emits `InvokeTypes` descriptors and can omit trailing `pythoncom.Missing` arguments; dynamic `CDispatch` may still synthesize typed calls from type information.
- The observed comtypes dynamic path is `lazybind.Dispatch`: it resolves members with `ITypeComp.Bind` and invokes an empty frame with null pointers. The no-type-information `_Dispatch` fallback uses `GetIDsOfNames`.
- comtypes generated code may switch from IDispatch to a dual-interface vtable binding, and its dynamic `Invoke` rejects arbitrary named keyword arguments.
- comtypes exposes explicit `VARIANT.missing`, `VARIANT.empty`, and `VARIANT.null` objects; pywin32 exposes `pythoncom.Missing` and special OLE marker objects.

The successful comtypes controls rule out neither an Excel/session difference nor a Rust defect; no rule here changes the Rust probe.
